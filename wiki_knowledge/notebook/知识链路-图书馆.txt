结构化路径： 用户问题 -> LLM 提取关键词/意图 -> 直接查询结构化目录/标签 
                -> 定位到具体的“章-节” -> 将该章节完整内容喂给 LLM。

① 目录索引查找 (Catalog Indexing)
将文档的目录结构存入关系型数据库或搜索树中。

做法： 先让 LLM 阅读目录，选择相关的章节 ID。

优势： 100% 准确，不会出现向量检索中常见的“明明就在眼前却搜不到”的情况。

② 长上下文窗口 (Long Context Window)
现在的 LLM（如 Gemini 1.5 Pro, Claude 3.5）支持 200k 甚至 2M 的上下文。

做法： 既然你的切分是按“章”来的，哪怕一章有 2 万字，你也可以一次性塞给 LLM。

优势： 彻底解决了“内容细碎”导致的上下文断裂问题，LLM 能看到全貌。

③ BM25 关键词检索 (Full-text Search)
这是一种传统的文本搜索算法（类似 Elasticsearch）。

做法： 利用精确的关键词匹配来辅助定位章节。

优势： 对于专有名词（如“MongoDB”、“OCR”），关键词匹配往往比向量匹配更可靠。


要实现从“自然语言”到“结构化 JSON”的精准转化，不能只靠一个简单的提问，而是需要一套被称为 Structured Output（结构化输出） 的技术方案。

在你的架构中，这应当被封装为一个名为 IntentAnalysisSkill 的核心能力。以下是具体的实现步骤：

1. 核心工具：System Prompt (系统提示词)
你需要为 LLM 设定一个极其严格的“身份”和“输出协议”。


# Role: 意图解析 Skill

用户正在与知识库对话，你需要作为翻译层，将自然语言转化为内部执行参数。

Step 2: 对比预定义的意图列表。
Step 3: 构建输出对象。

## Output Format
```json
{"intent": "...", "params": {...}}

## Few-Shot + Chain of Thought (思维链范式)
这种方式不只告诉模型“做什么”，还通过**例子**和**思考过程**告诉模型“怎么想”。这在处理复杂路由（比如判断到底该去哪个章节）时非常有效。

* **实现方式**：在 System Prompt 中加入 2-3 组 `Input -> Thought -> Output` 的示例。
* **效果**：能够极大降低模型在面对模糊问题时的“幻觉”和“胡乱分配意图”。



2. 从提取结果到“目录寻址”的闭环
得到这个 JSON 后，你的代码逻辑（也就是你架构中的 Capability Router）将执行以下动作：

-关键词召回：拿着 entity: "MongoDB" 去 Meilisearch（BM25）里搜，锁定文档范围。

-目录树定位：

此时你手中有一个 topic: "sharding"。

你去 MongoDB 的 catalog_tree 集合中，通过模糊匹配或语义相似度寻找标题中包含“分片”或“sharding”的 node_id。

-精准路由：

如果找到了 3.3 分片集群配置 这一章，直接跳过全量向量检索，精准读取该章节的 Block 块。

层级切分和目录树存储下的Agent 寻数据的逻辑会演变为以下三个步骤：

第一步：定位——从参数到 Node_ID
通过之前的“意图提取”和“参数化映射”，Agent 手里现在有一张“导航图”（JSON 参数）。它首先要去 MongoDB 的 catalog_tree（目录树集合）中找到那个“坐标”。

逻辑： Agent 拿着参数里的 entity（如 MongoDB）和 topic（如 分片/sharding）去查询目录树索引。

MongoDB 查询示例：

JavaScript
// 寻找标题中包含关键词且属于目标文档的节点
db.catalog_tree.find({
    "doc_id": "doc_001",
    "title": { $regex: /分片|sharding/i }
})
结果： 获得一个或多个 node_id（例如：node_ch3_sec3）。

第二步：提取——从 Node_ID 到 Content
一旦锁定了 node_id，Agent 就不再需要向量计算，而是直接通过物理关联去拿具体的内容块（Chunks）。

关联查询： 在你的设计中，每个文本块（Chunk）都存储了它所属的 node_id。

MongoDB 查询示例：

JavaScript
// 按照 block_index 顺序提取该节点下的所有文本块
db.document_chunks.find({
    "node_id": "node_ch3_sec3"
}).sort({ "block_index": 1 })
结果： 获得该章节完整、连续的 Markdown 文本。

第三步：还原——上下文增强（向上合并策略）
这是你架构中最体现“智能”的地方。如果 Agent 发现定位到的节点内容太少，或者用户问的是“前置条件”，它会触发向上溯源。

逻辑： 通过 parent_id 字段，向上查找父节点（章）或同级节点（前一节）。

实现：

获取当前节点的 parent_id。

拉取父节点的 summary（摘要）或者前一个节点的末尾内容。

拼接： 将“父级背景 + 当前节内容”合并，喂给 LLM。

总结：Agent 找数据的三种模式
根据你的 Skill 路由配置，Agent 可以切换不同的 MongoDB 查找模式：

精确模式 (Exact Path)：

路径： 意图解析 -> 提取面包屑路径 -> db.chunks.find({"breadcrumb": "章>节"})。

适用： 用户明确要求查看某个章节。

语义模式 (Semantic Metadata)：

路径： 意图解析 -> 提取关键词 -> db.chunks.find({"keywords": {"$in": ["分片"]}})。

适用： 目录匹配不明显，但在预处理阶段 LLM 注入了相关关键词。

全量召回模式 (Bulk Load)：

路径： 定位到“章” ID -> 拉取下面所有的“节”和“块”。

适用： 复杂任务，需要 LLM 对整章进行总结或对比。



文档知识库优化方案：层级结构化切分与智能路由检索
1. 结构化切分阶段 (Hierarchical Structural Chunking)
该阶段的核心是弃用“字符计数”的盲目切分，转而利用文档原始逻辑。

一级切分（代码硬逻辑）：

通过 Markdown 解析器识别标题层级（# 至 ###）。

将文档拆解为“章-节-小节”的树状结构块。

对于正常长度小节： 直接按目录层级存储。

对于超长小节（> 5000字）：

分块： 按 1500 字 + Overlap 策略切分。

语义注入： 对每个块，利用 LLM 自动提取该块的 Summary（摘要）和 Key_Keywords（关键词）。

存储： 在数据库里，这个块的检索权重是：标题路径 > 摘要 > 关键词 > 原文内容。

注意：对于语义注入产生的摘要，要有对原文的引用（用来标识是从原文哪里提取出来的）

元数据增强：

每个块（Chunk）在存储时必须强制关联其所属的标题路径（面包屑）及在目录树中的 Parent_ID。

2. 存储设计 (Knowledge Tree Storage)
为了支撑后续的智能路由，存储需要兼顾“检索”与“逻辑”双重需求。

结构化存储 (MongoDB)：

存储完整的文档目录树（Tree Structure）。

每个 Chunk 记录其在树中的位置。
存储架构：树状逻辑存储
存储媒介： 采用支持层级索引的数据库（如 MongoDB 或带有语义索引的传统库）。

数据模型： 每个内容块（Chunk）不再是孤岛，而是知识树上的一个叶子节点。必须包含 Node_ID, Parent_ID, Level（层级）以及 Path（全路径名）。
3. 智能路由检索阶段 (LLM-Based Intent Routing)
这是摆脱传统向量检索局限性的核心。检索不再是简单的“算距离”，而是“翻目录”。

第一步：意图与关键词提取：

利用 LLM 分析用户问题，提取核心实体与意图。

第二步：多级路由定位 (Routing)：

粗过滤： 通过关键词在 MongoDB 目录索引中筛选候选文档。

精定位： 将候选文档的**目录树（不含正文）**喂给 LLM，让 LLM 充当“图书管理员”，从逻辑上判断问题对应的具体章节 ID。

第三步：按需加载与上下文还原：

系统根据路由定位的 ID，从 MongoDB 直接提取该章节及其父级/同级章节。

优势： 这种方式绕过了向量搜索可能带来的“语义漂移”，确保给到 LLM 生成答案的内容是逻辑连贯的完整章节。

4. 方案核心优势
检索一致性： 检索结果与切分策略解耦。因为检索是基于目录结构的逻辑定位，不会因切分碎块的变动而导致结果南辕北辙。

高关联性： 通过智能路由获取的是“一整节”或“一整章”，LLM 拥有了真正的上下文，彻底解决细碎文档无法回答全局问题的痛点。

效率与精度平衡： 解析阶段虽然引入了 LLM 二级切分，但在检索阶段通过目录路由极大缩小了扫描范围，提高了整体系统的吞吐量。
